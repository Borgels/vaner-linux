# Maintaining vaner-desktop

Operational runbook for cutting releases, the apt-repo, and key rotation. Code changes belong in PRs; this file is the human-readable side of the release pipeline already wired up in `.github/workflows/release.yml` and `scripts/ci/`.

## Releases at a glance

A release is what happens when a `v*.*.*` tag is pushed. The workflow:

1. Builds Linux (`.deb` + `.AppImage`) on `ubuntu-22.04` and Windows (`.exe`) on `windows-2022`.
2. Signs every artifact with the GPG release key (`dpkg-sig` on `.deb`, detached `.asc` on the rest, signed `SHA256SUMS`).
3. Verifies all signatures before any upload — `scripts/ci/verify-artifacts.sh` is the gate.
4. Composes `latest.json` (the Tauri-updater manifest, distinct from the apt repo) with minisign signatures over the AppImage and `.exe`.
5. Publishes the GitHub Release.
6. Refreshes the apt repo on the `gh-pages` branch via `reprepro` and pushes.

Pushing `workflow_dispatch` runs steps 1–4 only — a dry run that proves the pipeline still works without actually cutting anything.

## The release signing key

There is one dedicated key used for all signing the apt repo cares about:

- **Identity:** `Vaner Release Signing (Vaner desktop .deb release key) <release@vaner.ai>`
- **Algorithm:** RSA-4096 (signing + cert)
- **Fingerprint:** `506B 8FA9 5991 7D53 0E5E E720 3D21 9B47 A7E4 F046`
- **Created:** 2026-04-24
- **Expires:** 2028-04-23
- **Public key:** committed at [`scripts/release-key.asc`](scripts/release-key.asc)
- **Pinned in:** [`scripts/install.sh`](scripts/install.sh) (`VANER_RELEASE_FPR`)

This key signs:

- The `.deb` (embedded `dpkg-sig` + detached `.asc`).
- The `.AppImage` and Windows `.exe` (detached `.asc` only — they have a separate Tauri-side minisign chain handled by `tauri-plugin-updater`).
- The apt repo's `Release` / `InRelease` files.
- The `SHA256SUMS` aggregate.

It does **not** sign the in-app updater payload — that's a separate minisign key (`TAURI_SIGNING_PRIVATE_KEY`) baked into `src-tauri/tauri.conf.json`. The two trust chains exist independently on purpose: someone replacing the apt repo metadata can't push a signed AppImage update, and vice versa.

## GitHub Secrets

The release workflow needs these secrets set on the `vaner-desktop` repo:

| Secret | Format | What it is |
|---|---|---|
| `VANER_RELEASE_GPG_PRIVKEY` | base64-encoded armored private key | The `release@vaner.ai` private key. Obtain via `gpg --armor --export-secret-keys release@vaner.ai \| base64 -w0`. |
| `VANER_RELEASE_GPG_PASSPHRASE` | plain text | The key's passphrase. |
| `VANER_RELEASE_GPG_FINGERPRINT` | 40-char hex, no spaces | `506B8FA959917D530E5EE7203D219B47A7E4F046` (the fingerprint above, with spaces stripped). |
| `TAURI_SIGNING_PRIVATE_KEY` | minisign private key text | Generated by `pnpm tauri signer generate` (Tauri docs). Used by `tauri-bundler` to sign the AppImage / NSIS for the in-app updater. |
| `TAURI_SIGNING_PRIVATE_KEY_PASSWORD` | plain text | Password for the minisign key. |

If `VANER_RELEASE_GPG_*` are missing, the `sign-artifacts.sh` step fails fast and the release does not publish. If `TAURI_SIGNING_*` are missing, the bundler still produces unsigned artifacts and the manifest step fails because there's no `.AppImage.sig`.

## First-time apt-repo bring-up

Once per repo lifetime — the workflow handles refreshes from then on.

1. **Set the four GitHub Secrets above.**
2. **Initialise the `gh-pages` branch.** The workflow's `initialize gh-pages if absent` step will do this automatically on the first tagged release. If you want to bootstrap manually:
   ```bash
   git checkout --orphan gh-pages
   git rm -rf .
   touch .nojekyll
   git add .nojekyll
   git commit -m "chore: initialize gh-pages for apt repo"
   git push origin gh-pages
   ```
3. **Configure GitHub Pages** in the repo's **Settings → Pages**: set the source to the `gh-pages` branch, root path. GitHub will serve the contents at `https://Borgels.github.io/vaner-desktop/`.
4. **Point the custom domain.** Add a `CNAME` file at the repo root of `gh-pages` containing `apt.vaner.ai`, then add a DNS `CNAME` record:
   ```
   apt   IN   CNAME   Borgels.github.io.
   ```
   Wait for GitHub to issue the TLS certificate (a few minutes), tick **Enforce HTTPS** in Pages settings.
5. **Cut the first release** by pushing a `v*.*.*` tag. The workflow signs artifacts, runs `build-apt-repo.sh` over `gh-pages`, and pushes. After it finishes:
   ```bash
   curl -fsSL https://apt.vaner.ai/release-key.asc | gpg --show-keys --with-fingerprint
   ```
   should print the fingerprint above.

## What an end-user install looks like

The recommended path is the SHA-pinned mirror at `vaner.ai/desktop.sh` (mirrored from `Borgels/vaner-desktop/main/scripts/install.sh` by [`vaner-web/scripts/fetch-install-script.mjs`](https://github.com/Borgels/vaner-web/blob/main/scripts/fetch-install-script.mjs)):

```bash
curl -fsSL https://vaner.ai/desktop.sh | bash
```

The mirror's prebuild step verifies the upstream SHA against the pin in `vaner-web/scripts/desktop-script.sha256`, so a tampered `main` fails the deploy rather than silently shipping. If you change `scripts/install.sh` here, bump the pin in vaner-web in the same release.

The raw GitHub URL (`https://raw.githubusercontent.com/Borgels/vaner-desktop/main/scripts/install.sh`) still works as a fallback if `vaner.ai` is unreachable, but documentation should point at the mirror.

Subsequent upgrades come through `apt upgrade` like any other package.

The script verifies the downloaded pubkey's fingerprint against the constant in `scripts/install.sh:36` before trusting any signature, so a swapped key fails with an explicit mismatch. For one-off installs without registering the apt repo:

```bash
VANER_MODE=deb curl -fsSL .../install.sh | bash
```

## Rotation procedure

The release key expires 2028-04-23. Plan to rotate at least three months before that.

1. **Generate the new key** in a clean GNUPGHOME, RSA-4096, expiry ~2 years out:
   ```bash
   GNUPGHOME=$(mktemp -d) gpg --batch --quick-generate-key \
     "Vaner Release Signing (Vaner desktop .deb release key) <release@vaner.ai>" \
     rsa4096 sign,cert 2y
   ```
2. **Export both halves**:
   ```bash
   gpg --armor --export release@vaner.ai > scripts/release-key.asc
   gpg --armor --export-secret-keys release@vaner.ai | base64 -w0 > /tmp/privkey.b64
   ```
3. **Update the fingerprint pin** in `scripts/install.sh` (`VANER_RELEASE_FPR`) and this file.
4. **Update the four GH Secrets**: `VANER_RELEASE_GPG_PRIVKEY` (paste contents of `/tmp/privkey.b64`), `VANER_RELEASE_GPG_PASSPHRASE`, `VANER_RELEASE_GPG_FINGERPRINT` (no spaces).
5. **Open a PR** with the new `scripts/release-key.asc` and pinned fingerprint. The release CI is gated on the pin matching the imported key, so a mismatched commit fails before any artifact upload.
6. **Cut a release.** The next tag rebuilds the apt repo with the new key signing `Release`. Existing users' apt clients will fail their next `apt update` until they re-import `release-key.asc`; document this in the release notes.
7. **Revoke the old key** (after the new one is verified working):
   ```bash
   gpg --output revoke.asc --gen-revoke <old-fingerprint>
   gpg --keyserver keys.openpgp.org --send-keys <old-fingerprint>
   ```
   Publishing the revocation is courtesy; the cutover already happened in step 6.

## Common operations

### Verify a downloaded `.deb` by hand

```bash
gpg --import scripts/release-key.asc
gpg --verify vaner_*.deb.asc vaner_*.deb
```

### Manually rebuild the apt repo locally

Useful when debugging `build-apt-repo.sh` failures:

```bash
mkdir -p /tmp/apt-staging
VANER_RELEASE_GPG_PRIVKEY="$(gpg --armor --export-secret-keys release@vaner.ai | base64 -w0)" \
VANER_RELEASE_GPG_PASSPHRASE="$your_passphrase" \
VANER_RELEASE_GPG_FINGERPRINT=506B8FA959917D530E5EE7203D219B47A7E4F046 \
  ./scripts/ci/build-apt-repo.sh /tmp/apt-staging path/to/your.deb
ls /tmp/apt-staging/dists/stable/main/binary-amd64/
```

The output mirrors what ends up on `gh-pages`. Inspect `Release.gpg` and `InRelease` for signatures.

### Drop a problematic release

```bash
git checkout gh-pages
reprepro -b . remove stable vaner
git commit -am "chore: yank vaner $version"
git push origin gh-pages
```

This deletes the package from the index but keeps the `.deb` in the GitHub Release for forensic / pinned-version use.

### Dry-run the workflow

Trigger `Release` from the **Actions** tab via `workflow_dispatch`. Steps 1–4 run; the upload + apt-repo refresh skip. Use this whenever you change `release.yml` or any `scripts/ci/*.sh`.
